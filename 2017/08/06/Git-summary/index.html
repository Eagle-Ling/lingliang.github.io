Git 命令

创建版本库

创建版本库

  • 创建一个空目录，目录名用英文 
    $ mkdir  //创建一个空目录
  $ cd  //进入目录
  $ pwd //该命令用于显示当前目录

  • 把目录变成Git可以管理的仓库（仓库初始化）

    git init

    把文件添加到Git仓库

  • 把文件提交到暂存区

    $ git add

  • 把文件提交到仓库

    $ git commit
      $ git commit -m “备注”
      $ git commit -am ‘xxx’ //将add和commit合并为一步

小结

添加文件到Git仓库，分两步：

  1. 使用命令git add <file>，注意，可反复多次添加多个文件；
  2. 使用命令git commit，完成。

时光机穿梭

查看工作区状态

$ git status

查看修改的内容

$ git diff <file>

工作区和暂存区的比较，当文件修改后还没提交之前，可用git diff命令查看修改的内容。

查看历史记录

$ git log
$ git log --pretty=oneline //简化输出信息

版本回退

$ git reset

例如，回退到上个版本：

$ git reset --hard HEAD^
`HEAD`表示当前版本，上一个版本是`HEAD^`，上上个版本`HEAD^^`，上10个版本写成`HEAD~100`。
例如，再回到当前版本： 
$ git reset --hard <版本id>

参数

–hard

–hard参数将会blow out everything.它将重置HEAD返回到另外一个commit(取决于~12的参数），重置index（暂存区）以便反映HEAD的变化，并且重置working copy（工作区）也使得其完全匹配起来。

这是一个比较危险的动作，具有破坏性，数据因此可能会丢失！如果真是发生了数据丢失又希望找回来，那么只有使用：git reflog命令了。所有本地修改将丢失!

如果我们希望彻底丢掉本地修改但是又不希望更改branch所指向的commit，则执行git reset –hard = git reset –hard HEAD. i.e. 另外一个场景是简单地移动branch从一个到另一个commit而保持index/work区域同步。这将确实令你丢失你的工作，因为它将修改你的work tree！

–soft

–soft参数告诉Git重置HEAD到另外一个commit，但也到此为止。如果你指定–soft参数，Git将停止在那里而什么也不会根本变化。这意味着index,working copy都不会做任何变化，所有的在original HEAD和你重置到的那个commit之间的所有变更集都放在stage(index)区域中。

–mixed

–mixed是reset的默认参数，也就是当你不指定任何参数时的参数。它将重置HEAD到另外一个commit,并且重置index以便和HEAD相匹配，但是也到此为止。working copy不会被更改。所有该branch上从original HEAD（commit）到你重置到的那个commit之间的所有变更将作为local modifications保存在working area中，（被标示为local modification or untracked via git status)，但是并未staged的状态，你可以重新检视然后再做修改和commit。

查看文件内容

$ cat <file>

查看每一次命令的记录

$ git reflog

小结

  • HEAD指向的版本就是当前版本，使用命令git reset --hard <id>可在各版本之间切换；
  • git log可查看提交历史；
  • git reflog查看命令历史。
  • 找回git add过但是已经不存在文件中的内容，用命令git fsck --lost-found。
  • 注意：如果在工作区修改了很多内容但没有add到并且commit就reset到之前的版本的话，那么这些修改就会消失，而且是无法挽回的！

撤销对工作区的修改

$ git checkout -- <file>

撤销对暂存区的修改

$ git reset HEAD <file

小结

  • 场景1：想直接丢弃工作区的修改，用命令git checkout -- file。
  • 场景2：想丢弃暂存区的修改，第一步用命令git reset HEAD <file>，就回到了场景1，第二部按场景1操作。
  • 场景3：已经提交了不合适的修改到版本库，用版本回退命令git reset --hard <版本id>
  • 如果已经提交到远程库……呵呵~

删除文件

  • 删除工作区的文件 
    $ rm <file>
  • 删除版本库的文件 
    $ git rm <file>

小结

  • 如果误删了工作区的文件，可用命令git checkout恢复，本质上是版本库里的版本替换工作区的版本。
  • 命令git rm用于删除一个文件。如果一个文件已经被提交到版本库，那么你永远不用担心误删，但是要小心，你只能恢复文件到最新版本，你会丢失最近一次提交后你修改的内容。

场景应用：假定你有若干电脑，你一会儿在公司提交，一会儿在家里提交，只要把每台电脑的Key都添加到GitHub，就可以在每台电脑上往GitHub推送了。

关联远程库

  1. 创建一个空的GitHub仓库；
  2. 在本地的仓库下使用命令：
    $ git remote add origin git@github.com:lingliang/<repository>

把本地内容推送到远程库

$ git push -u origin

-u参数的作用：第一次推送到master分支后，以后的推送就可以简化命令，直接git push。

从远程库克隆

$ git clone git@github.com:lingliang/<repository>.git

小结

  • 要克隆一个仓库，首先必须知道仓库的地址，然后使用git clone命令克隆。
  • Git支持多种协议，包括http，但通过ssh支持的原生git协议速度最快。

分支管理

分支创建与合并

示例：

  1. 创建dev分支，然后切换到dev分支。
    $ git checkout -b dev

    git checkout命令加上-b参数表示创建分支并切换；等同于：

    $ git branch dev
    $ git checkout dev

    git branch命令可查看当前分支。

  2. 把dev分支合并到master分支。
  • 先切换到master分支：git checkout master
  • 合并dev分支
    $ git merge dev
    删除`dev`分支 
    $ git branch -d dev

小结

  • 查看分支： git branch
  • 创建分支： git branch <name>
  • 切换分支： git checkout <name>
  • 创建+切换分支： git checkout -b <name>
  • 合并某分支到当前分支： git merge <name>
  • 删除分支： git branch -d <name>
  • 分支合并时，git会用Fast forward模式，但这种模式下，删除分支会丢失分支信息。如果加上--no-ff参数（表示禁用Fast forward），git会在合并时创建一个新的commit，这样就可以在分支历史上看见分支信息。

    合并分支时，加上--no-ff参数就可以用普通模式合并，合并后的历史有分支，能看出来曾经做过合并，而fast forward合并就看不出来曾经做过合并，相当于HEAD指向了dev分支 最新commit

Bug分支

$ git stash //储存工作现场
$ git stash list //查看储存的工作现场记录
$ git stash apply //恢复工作现场
$ git stash drop //删除stash的工作
$ git stash pop //恢复并删除stash的工作

修复bug时，我们会通过创建新的bug分支进行修复，然后合并，最后删除；

当手头工作没有完成时，先把工作现场git stash一下，然后去修复bug，修复后，再git stash pop，回到工作现场。

强制删除没有合并过的分支

$ git branch -D <name>

查看远程库信息

$ git remote
或 $ git remote -v //查看更详细的信息

推送分支

$ git push origin <branch>

抓取分支

$ git clone git@github.com:lingliang/<repository>.git

创建远程的分支到本地

$ git checkout -b <branch> origin/<branch>

建立本地分支与远程分支的联结

git branch --set-upstream <branch> origin/<branch>

多人协作模式

  1. 首先，可以试图用git push origin <branch>推送自己的修改；

  2. 如果推送失败，则因为远程分支比你的本地更新，需要先用git pull试图合并；

  3. 如果合并有冲突，则解决冲突，并在本地提交；

  4. 没有冲突或者解决掉冲突后，再用git push origin <branch>推送就能成功！

  5. 如果git pull提示“no tracking information”，则说明本地分支和远程分支的链接关系没有创建，用命令git branch --set-upstream <branch> origin/<branch>。

标签管理

创建标签

$ git branch <branch> //切换到需要打标签的分支
$ git tag <name> //打标签（默认标签是打在最新提交的commit上的）
$ git tag //查看所有标签
$ git tag <name> 'id' //对id对应的操作打标签

标签不是按时间顺序列出，而是按字母排序的。
git show <tagname>可查看标签信息

$ git tag -a <tagname> -m "description" //对标签进行说明

-a指定标签名，-m指定说明

操作标签

$ git tag -d <tagname> //删除标签

推送标签到远程

$ git push origin <tagname>
$ git push origin --tags //推送全部标签到远程

删除远程标签

1. $ git tag -d <tagname>
2. $ git push origin :refs/tags/<tagname>

自定义Git

让Git显示颜色

$ git config --global color.ui true

配置命令别名

$ git config --global alias.<别名> <命令>
如 $ git config --global alias.st status

git st作用等同于git status

命令$ cat .gitconfig可查看别名

配置Git的时候，加上--global是针对当前用户起作用的，如果不加，那只针对当前的仓库起作用。

每个仓库的Git配置文件都放在.git/config文件中，要删除别名，找到.gitcongig文件中删除。

忽略特殊文件

在Git工作区的根目录下创建一个特殊的.gitignore文件，然后把要忽略的文件名填进去，Git就会自动忽略这些文件。

忽略文件的原则是：

  1. 忽略操作系统自动生成的文件，比如缩略图等；
  2. 忽略编译生成的中间文件、可执行文件等，也就是如果一个文件是通过另一个文件自动生成的，那自动生成的文件就没必要放进版本库，比如Java编译产生的.class文件；
  3. 忽略你自己的带有敏感信息的配置文件，比如存放口令的配置文件。

举个例子：

假设你在Windows下进行Python开发，Windows会自动在有图片的目录下生成隐藏的缩略图文件，如果有自定义目录，目录下就会有Desktop.ini文件，因此你需要忽略Windows自动生成的垃圾文件：

# Windows:
Thunbs.db
ehthumbs.db
Desktop.ini

最后一步就是把.gitignore也提交到Git，就完成了！当然检验.gitignore的标准是git status命令是不是说working directory clean。

有些时候，你想添加一个文件到Git，但发现添加不了，原因是这个文件被.gitignore忽略了

$ git add -f <file>

就可强制添加到Git。
或者你发现，可能是.gitignore写得有问题，需要找出来到底哪个规则写错了，可以用git check-ignore命令检查：
